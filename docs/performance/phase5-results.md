# Phase 5 advanced execution modes

Phase 5 adds two explicitly opt-in execution experiments without changing the default
engine: bounded hot-function promotion inside a normal Broiler realm, and a separate
numeric bytecode/interpreter path that can be compiled offline into a Native AOT host.
It also records an isolated eight-byte tagged-scalar feasibility result. None of these
changes replaces the generic JavaScript slow paths or changes JSValue storage.

## Implementation matrix

| Delivery item | Implementation |
| --- | --- |
| Hot-function counters and delegate replacement | FunctionTieringController owns per-realm counters and budgets. Eligible JSFunction instances count invocations, reserve budget once, compile synchronously at the threshold, and atomically replace their invocation delegate. FunctionTieringSnapshot exposes attempts, successes, failures, rejections, replacements, deoptimizations, and retained-code estimates. |
| Bounded recompilation | Hosts opt in through UseFunctionTiering. Defaults are disabled, threshold 64, at most 32 recompilations, and at most 4 MiB of estimated retained code per realm. A rejected or failed site remains on its original delegate and does not retry. Generic eligible functions recompile through a private one-entry code cache. |
| Guarded numeric locals and deoptimization | NumericLoopPlanner recognizes a deliberately small counted affine-reduction shape. The promoted delegate guards the limit argument as a Number, keeps induction and accumulator values as unboxed doubles, and calls the captured baseline delegate when the guard fails. The canonical safe-integer sum from zero to n minus one uses an exact closed form only while its result remains inside Number's precise-integer range; general floating-point reductions are not reassociated. |
| Tagged-value feasibility prototype | Internal TaggedValuePrototype is an eight-byte NaN-boxed scalar for Number, Boolean, null, and undefined. It canonicalizes NaN and rejects objects, strings, symbols, and BigInts instead of inventing a GC-handle policy. It is not referenced by JSValue. |
| Offline/AOT execution | Broiler.JavaScript.Portable.Compiler parses and lowers a documented numeric subset ahead of deployment. Broiler.JavaScript.Portable validates and interprets immutable bytecode without reflection or dynamic code. The Native AOT sample references only the portable runtime and executes checked-in generated bytecode. |

## Tiering policy and semantic boundary

Tiering is off unless the host supplies an enabled FunctionTieringOptions instance.
The controller belongs to a JSContext, so counts and code budgets are not shared
between realms. JSContext.Eval and top-level-await evaluation now enter their target
realm explicitly; a regression keeps enabled and disabled contexts alive together and
proves compilation is charged to the target context.

The current candidate set is intentionally conservative:

- ordinary, non-arrow, non-class, synchronous user functions;
- no direct-eval compilation, with boundary, nested function, debugger statement,
  or dynamic scope;
- no capture from an outer function;
- locals must already satisfy the Phase 3 scalar-replacement observability checks.

Functions outside that boundary continue using the original delegate and do not
consume a tiering budget. Capturing closures and with functions are covered by a
zero-candidate regression. The generic tier recompiles capture-free source; the
numeric tier recognizes exactly one accumulator declaration, one counted for loop,
one affine += reduction, and a return of that accumulator.

Promotion happens on the invocation that reaches the configured threshold. Budget
reservation is locked per realm, compilation failure is swallowed in favor of the
baseline function, and delegate reads/writes are volatile. The retained-code value is
an intentionally conservative source-size estimate, not a runtime heap measurement.
It provides a stable host budget until compiled-code accounting is available.

## Numeric benchmark and deoptimization

The local Windows 11 x64/.NET 10 BenchmarkDotNet smoke job used one launch, three
warmups, and three measurement iterations. Setup promotes the tiered function outside
the timed operation and asserts that the disabled realm has zero candidates and the
enabled realm has exactly one successful recompilation.

| Input to sum(20,000) | Baseline | Tiered | Tiered / baseline |
| --- | ---: | ---: | ---: |
| Number | 743.954 us | 296.3 ns | 0.00040 (about 2,510x faster) |
| String "20000" | 631.634 us | 671.529 us | 1.06 |

The numeric result exercises the exact safe-integer reduction fast path. The string
case deliberately fails the Number guard and executes the original delegate; its
roughly 6% difference is inside the wide confidence interval of this short smoke run
and shows no mixed-type performance cliff. The semantic regression verifies
45|45|45|6|true|45 across baseline, promoted, repeated, fractional-numeric,
negative-zero, and string-deoptimized calls and
records exactly one delegate replacement and one deoptimization.

The budget regression creates three hot functions with a one-recompilation realm
budget. It observes one attempt and success, at least two budget rejections, and a
retained-code estimate no greater than the configured 4 KiB ceiling.

Run the focused benchmark with:

~~~powershell
$env:BROILER_BENCHMARK_PROFILE = "smoke"
dotnet run --project Broiler.JS/benchmarks/Broiler.JavaScript.Engine.Benchmarks -c Release -- --filter "*Phase5TieringBenchmarks*" --join
~~~

## Tagged-value feasibility result

Marshal.SizeOf<TaggedValuePrototype>() is 8 bytes. Scalar round trips cover Int32,
non-integer double, negative zero, NaN, Boolean, null, and undefined; an object
conversion is rejected.
The separate scalar-read smoke benchmark was effectively neutral:

| Representation, 1,024 scalar reads | Mean | Per-operation allocation |
| --- | ---: | ---: |
| Reference-backed JSValue array | 752.5 ns | 0 B |
| Inline eight-byte tagged array | 742.2 ns | 0 B |

This is a storage-density feasibility result, not a case for replacing JSValue.
Reference identity, moving-GC interaction, strings, symbols, BigInt, property storage,
interop, and debugger representation remain unresolved. The prototype therefore stays
internal and disconnected from production value storage.

## Portable language and host surface

The offline compiler accepts exactly one ordinary function with simple, unique
identifier parameters. All values are IEEE-754 doubles. Its supported syntax is:

- numeric literals, parameter reads, var local reads, and local assignment;
- +, -, *, /, %, numeric comparisons, and numeric equality/inequality;
- prefix/postfix increment and decrement plus arithmetic compound assignments;
- blocks, if, while, counted for, and explicit value-return statements.

Bytecode construction validates opcodes, argument/local indexes, jump targets,
operand-stack depth across control-flow joins, maximum stack, and reachable fallthrough.
The interpreter treats zero and NaN as false and accepts/returns only doubles.

The portable surface does **not** include the JavaScript object model, strings,
properties, arrays, functions or calls, closures, exceptions, host callbacks, eval,
modules, async/generators, reflection, or runtime compilation. Unsupported syntax is
rejected by the offline compiler. This is an alternate capability mode for numeric
programs, not a Native AOT build of the full Broiler engine.

The sample workflow is:

1. Compile fibonacci.js offline with Broiler.JavaScript.Portable.Compiler.
2. Emit FibonacciProgram.g.cs and compare normalized generated text with the
   checked-in file.
3. Publish Broiler.JavaScript.NativeAotSample, whose only project dependency is
   Broiler.JavaScript.Portable.
4. Execute the native binary and require Fibonacci(35) = 9,227,465 plus
   RuntimeFeature.IsDynamicCodeSupported == false and
   RuntimeFeature.IsDynamicCodeCompiled == false.

The local win-x64 report produced 5 files totaling 7,514,468 bytes. The native host
exited successfully with:

~~~json
{"program":"fibonacci","result":9227465,"dynamicCodeSupported":false,"dynamicCodeCompiled":false}
~~~

Regenerate the ignored report and publish directory with:

~~~powershell
python scripts/performance/collect_phase5.py --output artifacts/phase5
~~~
